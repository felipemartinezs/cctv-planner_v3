CCTV Field Planner v3

1. Qué es

Una Progressive Web App (PWA) diseñada para iPhone, desplegada en Google App Engine, que convierte un PDF de proyecto CCTV en herramientas operativas para técnicos de campo. Corre 100% en el navegador — no existe servidor de datos, no se envía información a ningún lugar.

2. El problema que resuelve

El problema que resuelve

Un técnico llega al sitio de instalación con un PDF que contiene:

  • Página 1: el plano arquitectónico con los IDs de dispositivos marcados numéricamente
  • Páginas 2+: tablas con datos de cada dispositivo (part number, switch, área, tipo)

Sin la app, el técnico tiene que navegar entre ambas partes del PDF, cruzar datos manualmente y llevar trazabilidad en papel. La app centraliza esto en una sola herramienta táctil.

5. Flujo de datos completo (cómo lo hace)

Flujo de datos completo (cómo lo hace)

5.1 Carga del PDF (src/lib/pdf.ts)

Función

loadPlan():

  • Lee el archivo como Uint8Array con readFileAsArrayBuffer
  • Abre el PDF con pdfjs-dist: getDocument({ data: buffer })
  • Extrae la Página 1 y llama a extractMarkers(page)

Extracción de marcadores (extractMarkers):

  • Obtiene el textContent de la página (todos los textos con sus posiciones exactas en coordenadas PDF)
  • Para cada TextItem con str que coincide con /^\d{1,3}$/ (número de 1 a 3 dígitos), captura:
  • x, y usando la matriz de transformación Util.transform([1,0,0,-1,0,height], item.transform)
  • El ID numérico
  • Maneja también el caso de múltiples IDs en un solo TextItem (ej: "45 46 47") distribuyendo posiciones proporcionalmente al ancho del texto
  • Resultado: Map<number, PlanMarker> con { id, x, y, label } — posición exacta de cada ID en el plano

Preview de imagen (renderPlanPreviewFromPage):

  • Detecta iPhone con /iPad|iPhone|iPod/i.test(navigator.userAgent)
  • En iPhone: renderiza a 1400px PNG, suavizado "high"
  • Hace page.render({ canvas, viewport }) → canvas.toBlob("image/png")
  • Devuelve previewUrl (blob URL) + dimensiones

5.2 Parseo de páginas de datos PDF (src/modules/pdf-data-parser/build-pdf-records.ts)

  • Lee las páginas 2+ del PDF con pdfjs
  • Reconstruye la tabla line por line usando tolerancias Y: LINE_Y_TOLERANCE = 2 (mismo renglón), ROW_Y_TOLERANCE = 15 (misma fila de tabla)
  • Agrupa fragmentos de texto en columnas (ID, Name, Part Number, Switch, etc.)
  • Detecta plantilla de datos automáticamente
  • Devuelve Record<string, string>[] — una fila por dispositivo

5.3 Normalización (src/lib/normalize.ts)

Función

normalizeRows(primaryRows, secondaryRows, markers):

Para cada fila de datos:

  • Extrae campos usando HEADER_ALIASES (maneja variantes: "switch" / "switch_name", "part_number" / "partnumber", etc.)
  • Asigna posición: si hay x, y en los datos → usa esa; si hay id → busca en markers del PDF; si no → hasPosition = false
  • Extrae área (extractArea): divide el nombre por _/, filtra prefijos conocidos (PTZ, POS, INT, EXT, etc.) y números puros → queda el área semántica
  • Categoriza (categorizeRecord): detecta PTZ, camera, monitor, mount, infrastructure por part number y nombre
  • Reglas de instalación (resolveInstallationSpec):
  • monitor interior → 8 ft
  • PTZ exterior → 16 ft
  • F360 en Receiving → 16 ft
  • F360 interior → 12 ft (necesita validación)
  • cámara exterior → 12 ft
  • Contextualiza icono (contextualizeIconDeviceForInstallation): PTZ outdoor → cambia a "CIP-QNP6250H Outdoor" automáticamente
  • Estima cables (estimateNetworkCables):
  • mount/infrastructure → 0
  • PVM compartida (Cosmetics, Sporting Goods, Baby Formula) → 0
  • Patrones especiales (CM-, HW-, NDA-, etc.) → 0
  • Regla por nombre → 1 o 2
  • Default → 1
  • Merge de duplicados: si el mismo ID aparece en fuente primaria y secundaria, fusiona los campos preservando el valor no vacío

Función

attachIcons(records, iconMap):

  • Consulta resolveRecordVisualKnowledge para seeds explícitos
  • Busca icono con lookupIcon (fuzzy match por part number → iconDevice → deviceTaskType)
  • Resolución final: exact match → score ≥ 5000 → vacío

5.4 Sistema de iconos (src/lib/icons.ts)

Tres fuentes de iconos (mergeadas):

  • Librería interna: cargada desde /device-icons/index.json (manifiesto pre-generado)
  • ZIP extra: descomprimido con JSZip, los PNG/SVG se codifican en base64 como data URLs
  • Folder extra: archivos via webkitRelativePath

Motor de búsqueda fuzzy (lookupIcon):

Score sistema (de mayor a menor):
- exact match          → 10000 + length
- query ⊂ candidate    → 8000 + ...
- candidate ⊂ query    → 7000 + ...
- common prefix ≥ 8    → 6000 + ...
- partial prefix       → 5000 + ...
MIN_FUZZY_ICON_SCORE = 5000

SVG tiene prioridad sobre PNG (iconPriority). El icono se descarta si no supera el umbral mínimo.

5.5 Segmentación espacial (src/modules/plan-segmentation/build-plan-segmentation.ts)

Función

buildPlanSegmentation(records, plan, gridWidth=170, kNeighbors=5):

  • Construye SegmentationPoint[]: solo dispositivos con id, posición y switch asignado
  • Crea una grilla de 170 × H celdas sobre el plano
  • Para cada celda, hace votación k-NN (k=5): busca los 5 puntos más cercanos y la etiqueta de switch más frecuente gana
  • Aplica suavizado de grilla (smoothGrid): cada celda toma el valor mayoritario de sus 8 vecinos (varios passes) → elimina ruido y produce zonas continuas
  • Produce PlanSegmentation con:
  • grid: Uint16Array — mapa de etiquetas por celda
  • segments[] — bounds, conteo de dispositivos, total cables por switch
  • partNumberTotals — conteo por part number
  • partNumberUnpositioned — dispositivos sin posición agrupados por part number
  • partNumberNoSwitch — dispositivos sin switch asignado

5.6 Vista de segmentación (src/modules/plan-viewer/PlanSegmentationModal.tsx)

Tres capas sobre el plano (stacked absolutamente):

  • <img> (no canvas) — la imagen PNG del plan renderizada desde pdfjs
  • overlayCanvasRef — pinta la segmentación coloreada por switch usando el Uint16Array de la grilla
  • partNumCanvasRef — dibuja círculos con IDs de dispositivos, marcadores pulsantes para el seleccionado

Pipeline de imagen en iPhone (3 fases):

  • Fase 1: plan.previewUrl (1400px PNG, disponible inmediato)
  • Fase 2: renderPlanPreview re-renderiza desde el PDF a 2600px PNG en background (async ~2-4s)
  • El <img> se actualiza cuando llega la fase 2

Zoom/pan:

  • react-zoom-pan-pinch con maxScale=8, minScale calculado al fit
  • gesturestart/change/end interceptados con preventDefault para bloquear el zoom nativo de Safari
  • touchAction: "none" en el contenedor del viewport

8. Por qué corre 100% en el cliente

Por qué corre 100% en el cliente

  • pdfjs-dist tiene su propio Web Worker (pdf.worker.min.mjs) que se descarga una vez y corre localmente
  • Los blobs del PDF se crean con URL.createObjectURL — la memoria es del navegador
  • localStorage guarda los estados de tarea
  • El service worker (PWA, vite-plugin-pwa) cachea todos los assets — funciona offline después de la primera carga
  • No hay ninguna llamada a una API externa en el código de procesamiento

9. Limitaciones actuales que el código refleja

Limitaciones actuales que el código refleja

  • plan.width = página 1 del PDF en user units (72 DPI). Si el PDF tiene una geometría no estándar, los marcadores pueden quedar fuera de bounds (se descartan en extractMarkers con y < 0 || y > viewportHeight)
  • La segmentación solo actúa sobre dispositivos con id numérico + posición + switch asignado — los que faltan quedan en partNumberUnpositioned o partNumberNoSwitch
  • Los estados de tarea (pending/active/done) se guardan en localStorage del dispositivo — si el técnico cambia de iPhone o limpia el caché, se pierden
  • La app no tiene autenticación (ningún código de auth en el repo)

Sección 10

El Motor de Inteligencia: Por qué lo hace y cómo lo hace

10.1 El problema raíz que resuelve esta pieza

Los PDFs de proyectos CCTV de Walmart tienen un problema estructural: los datos de dispositivos vienen incompletos o ambiguos. Un mismo dispositivo puede aparecer como:

  • "PTZ_APPAREL_WOMENS_47" — sin part number
  • "POS_COSMETICS_12" — nombre que no revela si es cámara domo o PTZ
  • "SAL_LIQUOR_PVM_8" — puede ser un monitor 32" o 43" back-to-back

Sin contexto de campo, el algoritmo no puede resolver esto solo con el texto del PDF. Aquí entra el sistema de conocimiento.

10.2 Arquitectura de tres capas

El motor de decisión está compuesto por tres capas que se aplican en orden de especificidad:

Capa A — namePatternRules (conocimiento por nombre de dispositivo)

Archivo:

src/config/manteca-visual-knowledge.json → namePatternRules

Qué hace: Mapea un patrón de nombre directamente a un part number específico y un icono específico, con un nivel de confianza.

Ejemplo real en el JSON:

json
{
  "namePattern": "PTZ EXT GRC",
  "suggestedPartNumber": "CIP-QNP6250H",
  "partConfidence": 1,
  "suggestedIconDevice": "CIP-QNP6250H Outdoor",
  "iconConfidence": 1
}

"Si el nombre del dispositivo es PTZ EXT GRC, el part number es con certeza absoluta (1.0) CIP-QNP6250H, y el icono correcto es el modelo Outdoor, no Ceiling ni Pendant."

Hay 70 patrones de nombre registrados — todos derivados del entrenamiento en campo del sitio Manteca.

Caso ambiguo registrado:

json
{
  "namePattern": "POS",
  "suggestedPartNumber": "BNB-SCB-1KIT",
  "partConfidence": 0.778,
  "candidatePartNumbers": ["BNB-SCB-1KIT", "PSA-W4-BAXFA51"],
  "iconConfidence": 0.556
}

"POS puede ser BNB-SCB-1KIT (77.8% de las veces) o PSA-W4-BAXFA51 (22.2%). El técnico debe validar en campo." → La app genera una advertencia visible para ese dispositivo.

Capa B — partNumberProfiles (conocimiento por part number)

Archivo:

src/config/manteca-visual-knowledge.json → partNumberProfiles

Qué hace: Cuando el part number sí viene en el PDF, define qué variantes de icono son posibles para ese part number.

Ejemplo:

json
{
  "partNumber": "CIP-QNP6250H",
  "iconDevices": [
    "CIP-QNP6250H Ceiling",
    "CIP-QNP6250H Outdoor",
    "CIP-QNP6250H Pendant"
  ]
}

Si el PDF dice CIP-QNP6250H y hay 1 sola variante posible → el icono se asigna automáticamente. Si hay múltiples → la Capa A (nombre) desempata, o si no puede → se genera advertencia.

Hay 14 part numbers perfilados actualmente.

Capa C — device-rules.ts (reglas de instalación con cables)

Archivo:

src/config/device-rules.ts

Qué hace: Agrega información que va más allá del icono — específicamente cuántos cables físicos requiere el dispositivo y qué nota debe ver el técnico.

Ejemplo:

device-rules.ts:45-54
{
  id: "liquor-pvm-43",
  description: "Monitor VPM 43\" en área Liquor (back-to-back, dual feed)",
  namePattern: "SAL_LIQUOR_PVM",
  inferredIconDevice: "MCLV-BAXFA51 43 LED Monitor 43 Back to Back",
  inferredPartNumber: "MCLB-BAXFA51",
  cablesCAT5: 2,
  cablesPower: 1,
  installerNote: "Monitor dual 43\" back-to-back — correr 2 CAT5 (uno por pantalla) + 1 cable 12/2 power"
}

Esto es conocimiento que no existe en ningún PDF — viene 100% del entrenamiento de campo.

Hay 6 reglas de instalación especializadas actualmente.

10.3 Cómo el algoritmo toma la decisión — flujo exacto

Para cada dispositivo,

resolveRecordVisualKnowledge en src/lib/visual-knowledge.ts sigue este árbol:

NOMBRE del dispositivo
       ↓
¿Hay namePatternRule con confidence = 1.0 y 1 solo candidato?
  → SÍ: part number + icono asignados con certeza total. FIN.
  → SÍ pero confidence < 1 o múltiples candidatos:
        Usa el sugerido + marca advertencia "validar en campo"
       ↓
¿El PART NUMBER del PDF está en partNumberProfiles?
  → El iconDevice del PDF ya coincide con uno en la lista:
        Confirmar — "existing-icon-device"
  → Solo hay 1 iconDevice posible para ese part:
        Asignar automáticamente — "part-number"
  → Múltiples opciones: ambigüedad → advertencia
       ↓
Fuzzy match final (src/lib/icons.ts)
  → Score mínimo 5000 para aceptar
  → SVG tiene prioridad sobre PNG
  → Si no supera el umbral: icono vacío

10.4 Por qué Manteca es la base y por qué importa

El archivo se llama manteca-part-number-04 porque fue construido a partir del proyecto Manteca — un sitio Walmart real donde se validaron en campo los 70 patrones de nombre contra los part numbers y las variantes de instalación reales.

Esto es la diferencia entre un algoritmo genérico y uno específico para Walmart:

Sin conocimiento Manteca Con conocimiento Manteca
PTZ_APPAREL_WOMENS → part number vacío, icono genérico PTZ_APPAREL_WOMENS → CIP-QNP6250H + icono Ceiling automático
SAL_LIQUOR_PVM → 1 cable estimado SAL_LIQUOR_PVM → 2 CAT5 + 1 power + nota instalador
POS_COSMETICS → ambiguo POS_COSMETICS → CIP-QND8011 con certeza 1.0

10.5 Cómo escala hacia SiteOwl

La arquitectura está diseñada para recibir múltiples seeds:

visual-knowledge.ts:24-26
export const VISUAL_KNOWLEDGE_SEEDS: VisualKnowledgeSeed[] = [
  mantecaVisualKnowledgeSeed as VisualKnowledgeSeed,
];

Para agregar SiteOwl como fuente de conocimiento, solo se necesita:

  • Crear siteowl-visual-knowledge.json con el mismo esquema (seedName, partNumberProfiles, namePatternRules)
  • Agregarlo al array: VISUAL_KNOWLEDGE_SEEDS = [mantecaKnowledge, siteowlKnowledge]

El motor en buildNamePatternRules() y buildPartNumberProfiles() mergea automáticamente los dos seeds, dando prioridad al de mayor partConfidence cuando hay conflicto.

SiteOwl tiene la información que elimina el 100% de las ambigüedades actuales (patrones ambiguos como POS con confianza 0.556) porque contiene la tabla completa de device_name → part_number → install_spec para todos los proyectos Walmart en su base de datos.
